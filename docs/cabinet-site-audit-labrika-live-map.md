## 4. Масштаб: сайт ~20 000 страниц

Опасения верные: **если хранить HTML всех страниц и все изображения как сущности — БД и краул раздуются**. На `torgmagvrn.ru` уже **~105k изображений** в счётчике — это сигнал не копировать модель «всё подряд».

### 4.1. Оценка времени краула

| Скорость (стр/с) | 5 000 стр | 20 000 стр | 50 000 стр |
|------------------|-----------|------------|------------|
| 1 | ~1.4 ч | ~5.5 ч | ~14 ч |
| 5 | ~17 мин | ~1.1 ч | ~2.8 ч |
| 10 | ~8 мин | ~33 мин | ~1.4 ч |

Реальность: robots delay, таймауты, JS-сайты, ретраи → закладывать **×2–3**.  
Вывод: краул 20k — **фоновые очереди** (как cluster/relevance), не синхронный HTTP из UI.

### 4.2. Оценка объёма БД

**Плохо (не делать):**

| Что | Оценка на 20k |
|-----|----------------|
| Полный HTML (~80–150 KB) | **1.5–3 GB** на один краул |
| HTML × 12 ежемесячных краулов | десятки GB на проект |
| 100k изображений с метаданными «в лоб» | сотни MB–GB |

**Нормально (целевая модель):**

| Таблица | На краул 20k | Комментарий |
|--------|--------------|-------------|
| `crawl_pages` (url, status, size, title, h1, canonical, content_hash, word_count, …) ~0.5–2 KB/стр | **10–40 MB** | без HTML |
| `crawl_findings` только ошибки ~0.3–1 KB × N | **1–20 MB** | sparse |
| `crawl_runs` агрегаты + 4 корзины | килобайты | для графиков |
| изображения: только **счётчики / топ-N проблем** | **≪ 1 MB** | не 105k строк обязательно |

Итого на проект при умной схеме: **десятки MB на краул**, не гигабайты.  
Историю детальных findings держать **N последних краулов** (например 3–6), старые — только агрегаты на график.

### 4.3. Что решает проблему

1. **По умолчанию не хранить HTML** в основной БД кабинета. Опциональные сжатые снимки — см. §4.4.  
2. **Findings = только отклонения**, не «OK по каждому из 200 факторов × 20k».  
3. **Инкрементальный краул:** сравнивать `content_hash` / etag, полный пересчёт дорогих факторов реже.  
4. **Лимиты тарифа:** страниц за краул + краулов/мес (как проверки/сохранения).  
5. **Очереди воркеров** + лимит RPS на домен.  
6. **Sitemap-first** + исключение фасетов/фильтров (настройки проекта).  
7. **Дорогие проверки (плагиат, PSI, все картинки)** — только на сэмпле / посадочных / по флагу.  
8. **Картинки:** агрегаты + «страницы без уникальных» без таблицы на 100k строк в MVP.

### 4.4. Инфра и опциональные HTML-снимки (решение)

**Целевая машина под краул:** отдельный хост (например `proxy2`: Ubuntu 24.04, много RAM/CPU), не shared VPS кабинета.

| Что | Где |
|-----|-----|
| UI / тарифы / юзеры | cabinet + основная MySQL |
| Очередь `site_audit` (fetch/parse) | remote worker host + supervisor |
| Метаданные аудита (`site_audit_*`) | желательно отдельная MySQL на remote (не загадить основную) |
| **Сжатый HTML** (если включено) | **только remote**: диск/том рядом с воркерами, не в основной БД |

**HTML-снимки (закладываем в дизайн):**

- Формат: **gzip** или **zstd** (`.html.gz` / blob), не сырой HTML в MySQL.  
- Включение **явное** (флаг run / настройка проекта / «сохранить снимок»), не по умолчанию на все URL.  
- Политики объёма: только findings / только посадочные / весь run в лимите тарифа; TTL + prune.  
- В `site_audit_pages` — путь или object key + `content_hash` + size_compressed, **не** сам файл в row основной БД.  
- Раздача в UI: cabinet запрашивает у remote API/storage по ключу (или signed URL), без копирования гигабайтов на app-сервер.

Оценка: gzip HTML часто **×5–10** меньше; 20k × ~15–30 KB сжатых ≈ **0.3–0.6 GB** на полный снимок run — терпимо на NVMe remote, недопустимо как бесконтрольный рост основной БД.

Рядом на том же хосте могут жить другие сервисы (напр. мобильные прокси): воркеры аудита не используют их proxy/маршруты модемов; исходящий default — обычный Ethernet.

## 5. Предложение MVP для Titlo (аналог, не клон 270 факторов)

### Фаза A — «Тех. lite» (быстро, дёшево по БД)

- Список URL: sitemap + ручной импорт (без глубокого BFS на старте).  
- По каждой: status, редирект-цепочка (лимит длины), size, title/description/h1, canonical, noindex, robots.  
- Отчёты: 4xx/5xx, редиректы, дубли title/desc, noindex, canonical «на чужой», большие страницы.  
- Сводка + график по `crawl_runs`.

**Переиспользовать:** http-headers, meta-tags, index-check (точечно).

### Фаза B — «SEO lite»

- Тощие страницы, блокировки индексации, потерянные посадочные (если есть список посадочных из мониторинга/ядра).  
- Похожие страницы (simhash) на сэмпле.  
- Сниппеты/каннибализация по сниппетам — стык с index-check (уже есть title+snippet).

### Фаза C — тяжёлое (позже / платные опции)

- Плагиат, взрослый/негативный контент, полный граф битых ссылок, все изображения, PSI mobile/desktop на весь сайт, аффилиаты.

### Лимиты (черновик)

| Тариф | Страниц / краул | Краулов / мес | Хранение findings |
|-------|-----------------|---------------|-------------------|
| Free | 500 | 1 | 1 краул |
| Optimal | 5 000 | 4 | 3 краула |
| Ultimate | 20 000 | 8 | 6 краулов |
| Maximum | 50 000+ | 12 | 6–12 |

(цифры обсуждаемые — зафиксировать после оценки железа.)

## 6. Модель данных (черновик)

```
site_audit_projects      (user_id, domain, settings_json)  -- save_html: off|findings|landing|all
site_audit_crawls        (project_id, started_at, finished_at, pages_total, counts_json)
site_audit_pages         (crawl_id, url, status, final_url, size, title, description, h1,
                          canonical, robots_flags, content_hash, word_count,
                          html_storage_key NULL, html_bytes_gz NULL, …)
site_audit_findings      (crawl_id, code, severity, url, meta_json)
site_audit_crawl_stats   (crawl_id, bucket, value)  -- для графиков
```

В базовой схеме **нет** `html_blob` в row. Опционально: `html_storage_key` → файл на remote (`…/{crawl_id}/{hash}.html.gz`).

Точка входа в кабинете: заменить заглушку `SiteAuditButtons` («В разработке»).

## 7. Решение «что делать»

| Вариант | Вердикт |
|---------|---------|
| Полный клон Labrika на 20k с HTML и всеми картинками | **Нет** по умолчанию |
| Краул сигналов + sparse findings + лимиты + очереди | **Да** — путь MVP |
| Опциональный gzip/zstd HTML **на remote worker host** | **Да** — по флагу, не в основной БД |
| Склейка существующих модулей без единого краула | Временно полезно, но не заменяет аудит |
| Начать с sitemap-листа ≤ лимита тарифа | **Да** — контроль объёма с первого дня |

**Итог:** MVP — сигналы и ошибки; сжатый HTML допускаем как **явную опцию на удалённой машине**, без засорения основной БД кабинета.
