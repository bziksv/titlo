# Кабинет — управление базой данных (changelog)

**Модуль:** `/admin/database`  
**Config:** `cabinet.datagon.ru/config/cabinet-database-admin.php`  
**Версия:** `1.1.3s`

## 1.1.3s — 2026-05-30

- **Просмотр** `search_indices` и других гигантских таблиц: `ORDER BY id DESC` (~1 с), не `created_at` (~70 с на 17M строк). Отдельное превью `search_indices` без url/snippet/title.
- **Проверка:** `/admin/database` → `search_indices` → «Просмотр» — панель за ~1–2 с, подсказка про id.

## 1.1.2s — 2026-05-29

- Кнопка **Очистить** только для `failed_jobs`: `TRUNCATE` + `OPTIMIZE`, затем обновление снимка инвентаризации. Whitelist `clearable_tables` в config — другие таблицы не трогаем.

### Проверка

1. http://localhost:3002/admin/database → фильтр по `failed_jobs`.
2. **Очистить** → подтверждение → 0 строк, размер ~0 MB.

## 1.1.1s — 2026-05-27

- Колонка **«Диапазон данных»** на русском: подписи полей (`created_at` → «Дата создания», `last_check` → «Последняя проверка»), даты в формате `дд.мм.гггг чч:мм`, «с … по …», бейдж **≈ по id** / **точно** вместо сырого `≈ id` и стрелки `→`.

## 1.1.0s — 2026-05-27

- Кнопка **Просмотр** у каждой таблицы: последние **10** строк (AJAX).
- `failed_jobs`: job, класс, краткая ошибка, фрагмент exception.
- Остальные таблицы: до 12 колонок, длинные поля обрезаются; тяжёлые blob исключены в config.

## 1.0.2s — 2026-05-27

- Крупные таблицы (>500 MB и список `date_probe_light_tables`): **лёгкий режим** — `ORDER BY id LIMIT 1` (секунды), не MIN/MAX по всей таблице.
- Для `relevance_history` дополнительно показывается `last_check` с последней строки.
- Бейдж **≈ id** в колонке «Диапазон данных».

## 1.0.1s — 2026-05-27

- Скан дат: **по 5 таблиц за клик**, последовательно (не параллельно, не все сразу).
- Пропуск таблиц **> 500 MB** и blocklist (`relevance_history`, `search_indices`, …).
- Исправлен баг кнопки: на странице отображалось `");"="">` (кавычки в `onsubmit`).

## 1.0.0s — 2026-05-27

- Пункт **«Управление базой данных»** в меню шестерёнки (`CabinetAdminMenu`).
- Страница инвентаризации: размер таблиц (`information_schema`), привязка к модулям (префиксы + `table_modules`), Eloquent-модели, число файлов с упоминанием таблицы, FK in/out.
- Фильтры: все / крупные (≥100 MB) / сироты / системные / без дат.
- Кнопки: **Обновить метаданные** (быстро), **Сканировать даты данных** (MIN/MAX по `created_at` и др., может занять минуты).
- Подсказка для `history_access` как вероятно забытая таблица.

### Проверка

1. Войти под admin → шестерёнка → **Управление базой данных**.
2. http://localhost:3002/admin/database — сводка ~118 таблиц, топ `relevance_history`, `search_indices`.
3. Фильтр **Сироты** — как минимум `history_access`.
4. **Сканировать даты данных** — в колонке «Диапазон данных» появляются min/max (подождать 1–3 мин).
